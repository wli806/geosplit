> 这是项目的"状态 + 决策"记录。每次会话开始先读:本文件 + `V2-架构.md` + `Auckland-GIS-数据接口手册.md`。

## 1. 产品定义(2026-06-22 方案调整)

输入奥克兰地址 → 自动产出:**GeoMap 多图层预览 + 3D 模型 + 分割收支平衡 + DEM/DSM/flood 成像**。将来网页收费。
- 目标用户:开发商/投资人/买家。
- **重大调整**:MVP **不做**自动房屋 layout(家/卧室/OLS/outlook 摆放)——太复杂、客户也不为半成品买单。layout 留作**以后的独立模块**。
- 产能只讲**理论上限**:根据坡度+分区,用启发式+经验公式**推荐分割成几块**,每块给面积/层数/透水率/室外面积。
- 房屋买卖/上市数据明确不在 MVP 范围。

## 2. 架构(已锁定,详见 `V2-架构.md`)

- 全新写 **V2**(`backend/` + `frontend/`;原 Stage 0 代码已改名并归档到 `backup/backend_stage0/`/`backup/frontend_stage0/`),**五层单向依赖**:sources → domain → render → api → frontend。
- 三条铁律:单向依赖;domain 出抽象 / render 出画法;数据带溯源。
- 2D = Leaflet,3D = Cesium;2D→3D 有"万丈高楼起"转场。
- 旧 Stage 0 代码只作参考,逐块搬过来、跑通一块删一块旧的。

## 3. 关键决策(摘要,完整见 `V2-架构.md` 第 5 节)

- **CV/RV**:官方 `Rates24_SnapshotReval` 即用即查(LCV/LLV 最新),经验值兜底;CV 当锚点不当卖价;商用授权待确认。
- **栅格**:tifffile + imagecodecs 读 E 盘 1m DSM/DEM,避开 GDAL。地形来源从 council 等高线**改为真 DSM/DEM**。
- **产能/收支**:`domain/subdivision.py` + `domain/economics.py` 两个独立模块。
- **数据溯源**:每个字段带来源/取数时间/是否本地,前端可标注、可做本地 vs 在线新鲜度对比。

## 4. 数据资产现状

- 公开接口:LINZ(地块/产权)、council(分区/overlay/管网/地质/洪泛/估值)全部已验证 → 见《接口手册》。
- 本地高程:`E:\Geomaps Data BASE\{DEM,DSM}\`,1m GeoTIFF(NZTM2000,LZW 压缩),约半个奥克兰,验证够用。
- 规则数值:MHU 全套已确认 → 见 `AUP-MHU规则数值.md`。

## 5. 规则范围与权威

- MVP 范围:**仅 MHU 分区 + fee simple 细分 + urban subdivision**;rural、复杂 overlay 暂不做。
- 规则/数值权威 = planner + 负责人,我负责结构化;不凭记忆编条款号或数字。
- 待 planner 确认的点见 `待确认-planner问题.md`。
- "围现有开发的细分"走 RD consent、不卡最小地块/shape factor —— 已确认,印证"先建后分"路径可行。
